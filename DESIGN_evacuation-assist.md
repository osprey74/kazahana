## 1. 概要とコンセプト

本機能は、kazahana が購読する BSAF ボット（bsaf-kikikuru-bot）の気象警報・注意報情報を活用し、ユーザーの現在地（都道府県）に警報級以上の情報が発令された際に、最寄りの避難所情報と避難誘導を「そっと差し出す」補助機能です。

設計の基本思想は次のとおりです。

- **主体は人間、アプリは一助**：避難の最終判断は利用者本人が行います。本機能は判断を代行せず、いざ避難する際の「避難所情報の取得」と「避難誘導」を補助することに徹します。
- **ガチガチに作らない「お節介機能」**：すべてを機械的に判定して自動作動させるのではなく、利用者が設定でオン／オフを選べる任意機能とします。「あって良かった」と感じてもらえる温度感を目標とします。
- **先回り型の動線設計**：警報級（レベル3）の段階で画面下部にバナーを常駐させ、避難所への動線を事前に用意します。状況がレベル4に進行した際、利用者はタップ一つで避難所情報・ナビに到達できます。プッシュの即時性に過度に依存しません。
- **オフライン耐性**：避難所データは端末に同梱し、通信途絶時でも最寄り避難所の方向・距離を表示できる簡易ナビを最終防衛線として持ちます。

---

## 2. スコープ

### 2.1 本機能で実現すること

1. 全国の指定緊急避難場所・指定避難所データを kazahana に同梱・保持する。
2. kazahana モバイル版に位置情報取得機能を追加する。
3. bsaf-kikikuru-bot のレベル3以上の情報が現在地（都道府県）に発令されたら、アプリ画面下部にバナーを表示する。
4. バナーから最寄り避難所情報を表示する／OSナビアプリへ遷移してナビを開始する／kazahana 内で避難所の方向・距離を表示する簡易ナビを行う。

### 2.2 本機能で実現しないこと（スコープ外）

- 自治体の避難指示（避難情報）そのものの配信。本機能が扱うのは気象庁の危険度・警報級情報（キキクル相当）であり、自治体の避難指示ではありません。
- Lアラート連携。個人 OSS では情報伝達者登録・契約の壁があるため対象外とします。
- 自前の経路探索エンジン。経路案内は OS 標準地図アプリへ委譲します。
- バックグラウンド常時測位による自動避難判定。

## 3. 全体アーキテクチャ

```
[気象庁 防災情報XML（VPWW55〜61）]
        │ 10分間隔ポーリング
        ▼
[bsaf-kikikuru-bot (Fly.io nrt)]
        │ BSAF v1 タグ付き投稿（type / value / time / target / source）
        ▼
[Bluesky / AT Protocol]
        │ 購読
        ▼
[kazahana モバイル版]
  ├─ BSAF購読・フィルタ（現在地都道府県 × level3以上）
  ├─ バナー状態管理（出現／昇格／解除）
  ├─ 位置情報取得（Tauri Geolocation Plugin）
  ├─ 同梱避難所DB（GeoJSON / SQLite）
  ├─ 最近傍探索（Haversine）＋災害種別フィルタ
  └─ ナビ（OS地図委譲 ／ コンパス簡易ナビ）
```

> **Note:** 初期実装はアプリ側ポーリングで成立します。将来的に即時性を高める場合、既存の kazahana push backend に BSAF監視＋地域別プッシュを追加する拡張余地を残します（第9章参照）。

## 4. Step 1：避難所データの保持

### 4.1 データソース

国土地理院「指定緊急避難場所・指定避難所データ」を使用します（災害対策基本法第49条の4・第49条の7に基づき市町村長が指定した施設を各市町村が登録したもの）。

- ダウンロード元: https://hinanmap.gsi.go.jp/index.html
- 含まれる情報: 施設名・住所・緯度経度・対応災害種別フラグ（洪水／崖崩れ・土石流・地滑り／高潮／地震／津波／大規模火事／内水氾濫／火山現象）

### 4.2 変換と同梱

- CSV → GeoJSON（または SQLite）に変換し、アプリにバンドルします。
- フィールドを「施設名・緯度・経度・災害種別フラグ」に絞り、サイズを最小化します（全国でも数MB程度を見込む）。
- オフライン動作を保証するため、ネットワーク非依存で読み込める形式とします。
- データ更新は起動時または定期チェックで差分確認する方式を将来的に検討します（初期はアプリ更新に同梱でも可）。

> **iOS 実装実績**: CSV → コンパクト JSON → zlib 圧縮で同梱。`id` フィールド除去、`hazards` を8ビットのビットマスクで表現し、全国 115,447件を 2.1MB に圧縮。変換スクリプトは `kazahana-ios/scripts/build-shelters.py`（Python）。Android でも同じ出力ファイルを使用可能。

### 4.3 留意点

- 国土地理院データは最新でない場合や未掲載施設がある旨が明記されています。アプリ内に出典と「最新情報は自治体へ」の注記を記載します。
- 災害種別フラグは後段の最近傍探索フィルタで使用するため、変換時に確実に保持します。

---

## 5. Step 2：位置情報取得

### 5.1 実装

- Tauri v2 公式 Geolocation Plugin を使用します（iOS / Android 対応）。
- iOS: `Info.plist` に `NSLocationWhenInUseUsageDescription` を記載。
- Android: `AndroidManifest.xml` に `ACCESS_FINE_LOCATION` / `ACCESS_COARSE_LOCATION` を記載。

### 5.2 測位ポリシー

- **バックグラウンド常時測位は行いません**（バッテリー・プライバシー・ストア審査の観点）。
- 都道府県の特定は、初回または設定画面での手動設定でも可とします（粒度は都道府県で足りるため）。
- 精密な現在地測位は「最寄り避難所を探す」操作時にのみフォアグラウンドで実行します。

## 6. Step 3：レベル3以上の検知とバナー表示

### 6.1 トリガー条件

bsaf-kikikuru-bot の BSAF タグを購読し、次の条件を満たす有効な投稿があるとき、バナーを表示します。

- `target` がユーザーの現在地都道府県と一致（例: `jp-hokkaido`）
- `value` が `level3` / `level4` / `level5`（または警報級体系の `warning` / `special-warning`）

### 6.2 バナーの状態遷移

```
[平常時] バナーなし
   │ 現在地都道府県に level3+ 投稿を検知
   ▼
[レベル3] 画面下部にバナー常駐
   「⚠️ ○○県に警報級の気象情報。最寄り避難所を見る ＞」
   │                      │
   │ level4/5へ進行         │ cancelled検知 or タイムアウト
   ▼                      ▼
[レベル4/5] バナー強調表示     [平常時] バナー消滅
   タップ → 避難所詳細 → ナビ
```

### 6.3 状態管理の要件

- BSAF の重複排除は `type + value + time + target` の完全一致です。レベル3→4への昇格、`cancelled`（解除）の検知を取りこぼさない実装とします。
- **解除の見逃し対策**：`cancelled` を受信できなかった場合に備え、一定時間（例: 最終受信から N 時間）で自動失効するタイムアウトを併設します。
- **複数現象の同時発令**：複数の現象が同時に level3+ の場合は、「複数の警報級情報」とまとめて表示するか、最高レベルの現象を代表表示します（表示ルールを別途定義）。

### 6.4 遅延の扱い

bsaf-kikikuru-bot は10分間隔ポーリングのため、最大10分超の遅延が生じ得ます。本機能はレベル3で先回り表示する設計のため、この遅延は許容範囲とします（避難フェーズではなく認知・準備フェーズでの表示のため）。

---

## 7. Step 4：避難所情報の表示と誘導

### 7.1 最寄り避難所の算出

- 同梱避難所DBに対し、現在地から Haversine 距離で最近傍を算出します。
- **災害種別フィルタを必ず適用します**。発令現象に対応する施設のみを候補とします（例: 大雨・洪水なら洪水／内水氾濫対応施設、土砂なら崖崩れ等対応施設）。
- 候補は最寄り1件だけでなく、近い順に複数件提示する選択肢も検討します。

### 7.2 ナビゲーション3方式

1. **避難所情報表示**: 施設名・住所・対応災害種別・直線距離を表示。
2. **OSナビアプリへ遷移**: URIスキームで目的地（緯度経度）を渡し、Apple Maps / Google Maps の徒歩ナビを起動します。自前経路エンジンは持ちません。
3. **kazahana 内簡易ナビ**: 現在地と避難所の方位角を計算し、デバイスのコンパス（磁気センサ）と組み合わせて「方向の矢印＋直線距離」を表示します。通信途絶時でも動作する最終防衛線です。

### 7.3 オフライン時の挙動

- 通信途絶時、OSナビの経路案内は使えません。
- その場合でも、同梱データとコンパスにより「最寄り避難所の方角・直線距離・地図上のピン」を表示できる簡易ナビを動作させます。

## 8. 設定とユーザー同意

- 設定画面に「避難誘導機能をオンにする／しない」のトグルを設けます（デフォルトの初期値は要検討。オフ初期＋初回案内が無難）。
- オンの場合のみ、位置情報取得・バナー表示が有効になります。
- 現在地都道府県の手動設定オプションを提供します。

> **iOS 実装実績**: デフォルトオフ。初回起動時にオンボーディングダイアログ（「避難誘導機能が利用できます。設定からいつでもオンにできます。」）を表示。有効化時に bsaf-kikikuru-bot の BSAF 自動登録確認ダイアログを表示。

---

## 9. 将来拡張（スコープ外・余地として記載）

- **即時プッシュ通知**：レベル4以上の即時性を高めたい場合、既存の kazahana push backend（Fly.io nrt + Bun + Hono）に「BSAFタグ監視＋該当都道府県ユーザーへの APNs/FCM プッシュ」を追加します。アプリ単体ポーリングからサーバ主導プッシュへの拡張です。
- **津波対応**：津波は bsaf-kikikuru-bot ではなく bsaf-jma-bot の担当です。津波トリガーを加える場合は購読先を追加し、避難所の津波対応フラグでフィルタします。
- **避難所開設・混雑情報**：将来的に外部API（全国避難所ガイド等、商用）連携の検討余地。ただしオフライン原則と要相談。

---

## 10. 免責・倫理・コンプライアンス

人命に関わる機能であるため、次を必須要件とします。

- **免責表示**：バナー近傍および設定画面に、簡潔かつ明確な一文を表示します。
  - 例:「この情報は気象庁の危険度情報（警報級）に基づく補助です。避難の判断は自治体の避難指示や公式情報をご確認ください。」
- **性質の明示**：本機能が扱うのは気象庁の警報級・危険度情報であり、自治体の避難指示そのものではないことを明示します。
- **データの注記**：避難所データは最新でない場合がある旨と、国土地理院を出典として記載します。
- **専門家確認**：リリース前に、表示文言・免責・責任範囲について専門家（弁護士、可能なら防災情報に明るい者）の確認を経ることを推奨します。

---

## 11. ストア審査の留意点

- 位置情報・通知・災害関連を扱うため、Apple / Google の審査で用途説明と免責を求められる可能性があります。
- 位置情報の利用目的（最寄り避難所の算出）を Privacy 説明に明記します。
- Michi-Navi での審査経験を活用します。

> **iOS 審査実績**: `NSLocationWhenInUseUsageDescription` に日英で目的を明記。バナー等のイベント駆動機能は審査中に再現できないため、隠しデモモード（バージョン番号5回タップ）で審査官が自分で操作できるようにした。

## 12. 技術スタック

| 領域 | 採用技術 |
|:--|:--|
| アプリ基盤 | Tauri v2 + React + TypeScript（kazahana既存） |
| 位置情報 | Tauri v2 Geolocation Plugin |

| ナビ委譲 | OS地図 URIスキーム（Apple Maps / Google Maps） |
| 将来プッシュ | kazahana push backend（Fly.io nrt + Bun + Hono + APNs/FCM） |

## 13. 実装フェーズ（HANDOFF向け叩き台）

- **Phase 1**: 避難所データ変換・同梱（Step 1）＋設定トグル
- **Phase 2**: 位置情報取得（Step 2）＋最寄り避難所表示・OSナビ委譲（Step 4の一部）
- **Phase 3**: BSAF購読・バナー状態遷移（Step 3）
- **Phase 4**: kazahana内簡易ナビ（コンパス）＋オフライン挙動（Step 4の残り）
- **Phase 5**: 免責文言確定・ストア審査対応・（任意）push backend拡張

> **iOS 実装状況**: 全 5 Phase を完了し v3.2.0 でリリース。専門家（弁護士・防災専門家）による免責文言確認済み。App Store 審査対応として、設定画面のバージョン番号5回タップで隠しデモモードを表示する方式を採用（Release ビルドでも審査官がアラートシミュレーション可能）。
