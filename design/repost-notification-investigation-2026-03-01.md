# リポスト経由の通知ナビゲーション — 調査記録 (2026-03-01)

## 背景

macOS版v2.0.1で「リポスト経由の通知クリックで投稿が見つかりません」が修正されていないとの報告を受け調査。

## 元のバグ (v2.0.1で修正済み — commit b6daa42)

- **症状**: 通知タブでリポスト経由の通知をクリック → 「投稿が見つかりません」と表示
- **原因**: `notification.reasonSubject` がリポストレコードURI（`at://did/app.bsky.feed.repost/rkey`）の場合、`getPostThread` がポストとして解決できない
- **修正内容**:
  1. `useSubjectPosts` フック: リポストURIを検出 → `getRecord` で元ポストURI取得 → ポストデータをマップに格納
  2. `NotificationItem.handleClick`: `targetUri` にリポストURIが含まれ `subjectPost` が解決済みなら元ポストURIに差し替え
  3. `NotificationList`: reason種別に応じた subjectPost lookup キー分岐

## v2.0.4で追加した防御処理 (commit 0381f0e)

- **問題**: `useSubjectPosts` のリポストURI解決（追加API呼び出し）が完了する前にユーザーがクリックすると、生のリポストURIが `useThread` に渡される（レースコンディション）
- **修正**: `useThread.ts` に `resolveRepostUri()` 関数を追加。`getPostThread` 呼び出し前にリポストURIを検出・解決するフォールバック処理

## Bluesky通知モデル

### Bluesky公式Web設定画面の通知項目（2026-03-02確認）

| 設定項目 | 説明 |
|---|---|
| いいね | 投稿をいいねされたら通知を受け取る |
| 新しいフォロワー | フォローされたときに通知を受け取る |
| 返信 | 投稿に返信があったら通知を受け取る |
| メンション | メンションされたら通知を受け取る |
| 引用 | 投稿を引用されたら通知を受け取る |
| リポスト | 投稿をリポストされたら通知を受け取る |
| リポストへのいいね | リポストした投稿をいいねされたら通知を受け取る |
| リポストのリポスト | リポストした投稿をリポストされたら通知を受け取る |

### 通知の宛先とreasonSubject

| アクション | 通知の宛先 | reasonSubject |
|---|---|---|
| BがAのポストをいいね | A（元ポスト作者） | AのポストURI |
| BがAのポストをリポスト | A（元ポスト作者） | AのポストURI |
| BがAのポストをリポスト → Cがいいね | A（元ポスト作者） | AのポストURI |
| BがAのポストをリポスト → Cがいいね | **B（リポスター）** | **BのリポストURI（推定）** |
| BがAのポストをリポスト → Cがリポスト | A（元ポスト作者） | AのポストURI |
| BがAのポストをリポスト → Cがリポスト | **B（リポスター）** | **BのリポストURI（推定）** |

**重要（2026-03-02修正）**: Bluesky公式設定に「リポストへのいいね」「リポストのリポスト」が存在することから、リポストしたユーザ（ユーザB）にも通知が届く仕様であることが確認された。この場合、`reasonSubject` にはリポストレコードURI（`at://did/app.bsky.feed.repost/rkey`）が設定されると推定される。これは通常のUI操作で発生する正規のルートである。

## テスト実施内容 (v2.0.4 — macOS + Windows)

### テスト1: 通常のいいね通知クリック
- **操作**: 通知タブで過去のいいね通知をクリック
- **結果**: 正常遷移。`targetUri` = `at://did/app.bsky.feed.post/...`（正しいポストURI）

### テスト2: リポスト通知クリック（メインアカウント投稿 → サブアカウントリポスト）
- **操作**: メインアカウントで投稿作成 → サブアカウントでリポスト → メインアカウントの通知をクリック
- **結果**: 正常遷移。`targetUri` = `at://did/app.bsky.feed.post/...`（正しいポストURI）

### テスト3: リポスト経由の通知再現試行
- **操作**: メインアカウントで他者のポストをリポスト → サブアカウントで同じポストをリポスト
- **結果**: メインアカウントに通知が届かない（Bluesky仕様通り — 通知は元ポスト作者にのみ届く）

### テスト3の補足（2026-03-02追記）

テスト3で通知が届かなかったのは、リポスターへの通知（「リポストへのいいね」「リポストのリポスト」）を再現するテストシナリオではなかったため。リポスターへの通知を再現するには、ユーザBがリポストした投稿に対して、第三者ユーザCがいいね or リポストする必要がある（元ポストの作者Aとは別のアカウントから）。

### 結論

- v2.0.4で「投稿が見つかりません」は再現せず（テスト範囲内）
- ~~`reasonSubject` にリポストレコードURIが入る条件は通常のUI操作では発生しにくい~~ → **訂正**: Bluesky公式設定「リポストへのいいね」「リポストのリポスト」が有効な場合、通常のUI操作で `reasonSubject` にリポストレコードURIが入る
- ~~元のバグ報告（あやがねさん）は特定のAPI挙動下で発生した可能性がある~~ → **訂正**: あやがねさんのバグ報告は、リポストした投稿に対して第三者がいいね/リポストした際の正規の通知ルートで発生していた可能性が高い
- v2.0.4の二重防御（useSubjectPosts + useThread resolveRepostUri）は正しい修正であり、この正規ルートを適切にカバーしている

## like-via-repost / repost-via-repost 対応（2026-03-02）

### サーバーサイド前提条件

- Bluesky公式Web設定で「リポストへのいいね」「リポストのリポスト」が**デフォルトOFF**であることを確認
- ユーザーが公式Webで設定をONにした後、新しいアクションに対してのみ通知が生成される（遡及なし）

### クライアント実装

以下のファイルに `like-via-repost` / `repost-via-repost` reason対応を追加:

- `NotificationItem.tsx`: アイコン（赤ハート/緑リピート）、ラベル、色、displayText分岐
- `useNotifications.ts`: `useSubjectPosts` のURI収集条件
- `lib/notifications.ts`: `NotificationReasonCounts` 型と `REASON_KEYS`
- `i18n/locales/en.json`, `ja.json`: 翻訳キー

### テスト4: like-via-repost / repost-via-repost（2026-03-02実施）

- **前提**: ユーザA（リポスター、通知設定ON）、ユーザB（Aのフォロワー）、ユーザZ（Bがフォローしていない）
- **操作**: ユーザZのポストをユーザAがリポスト → ユーザBのTLに「ユーザAがリポスト」で表示 → ユーザBがいいね＋リポスト
- **結果**:
  - ユーザAのKazahana通知タブに `like-via-repost` / `repost-via-repost` 通知が表示
  - アイコン（赤ハート/緑リピート）とラベル（「があなたのリポストをいいねしました」等）が正しく表示
  - 通知クリックでユーザZの元ポストに正しく遷移

### 補足: 通知が届かなかった初回テストの原因

- ユーザAのBluesky設定で「リポストへのいいね」「リポストのリポスト」がOFFだった
- ONにした後もいいね/リポストをやり直す必要があった（設定変更前のアクションには遡及されない）
